# Sparkle auto-updates for VibePulse

**Status:** Draft for review
**Date:** 2026-04-13
**Author:** Wes McKinney (with Claude)

## Goal

Add in-app automatic update delivery to VibePulse so users receive signed,
notarized new versions without manually downloading DMGs from GitHub Releases.

Ship via [Sparkle 2.x](https://sparkle-project.org/), the de facto Mac
auto-update framework. Leverage the existing signed + notarized DMG pipeline
in `.github/workflows/release.yml`; add appcast generation and framework
embedding rather than building a second release path.

## Non-goals

- Delta updates (binary diffs). Full DMG each release is fine at VibePulse's
  size (~275 KB).
- Beta / prerelease channels. Single stable feed only.
- Rollback UI. Sparkle's own mid-install rollback is sufficient.
- Self-hosted update server. GitHub Releases hosts the DMG and appcast.
- In-app changelog browsing. Release notes render in Sparkle's built-in
  dialog; GitHub is the source of truth.

## User experience

**Standard Sparkle prompt.** When a new version is found, Sparkle shows its
native dialog with rendered release notes and **Install Update**, **Remind
Me Later**, and **Skip This Version** buttons. On **Install**, Sparkle
downloads the DMG, verifies the EdDSA signature, replaces the app bundle via
its XPC helper, and relaunches into the new version.

**Check cadence.** When automatic checks are enabled (the default from
`SUEnableAutomaticChecks=true`), a silent background check runs shortly
after app launch — triggered explicitly by `UpdaterController.init`
only when `updater.automaticallyChecksForUpdates` is true, so a user
who has disabled automatic checks is not overridden. Sparkle's
scheduler then runs a further background check every 24 hours via
`SUScheduledCheckInterval`. For a menubar app that typically runs
24/7, the on-launch check fires once per cold start and the scheduler
carries the rest. See the *Explicit on-launch check (guarded)* item
under *Architecture → Client* for why this is explicit rather than
implicit and why it is gated on the persisted preference.

**Manual trigger.** A "Check for Updates…" button is added to the menubar
popover controls row (next to Refresh, Settings, Quit). Clicking it runs a
user-initiated check; Sparkle shows a "You're up to date" dialog when there
is nothing newer.

## Architecture

### Client (app bundle)

- **Sparkle 2.x** added as a SwiftPM dependency in `Package.swift`, pinned to
  an exact version (e.g. `.exact("2.6.4")`).
- New file `Sources/VibePulse/Services/UpdaterController.swift` — thin
  wrapper around `SPUStandardUpdaterController`. Exposes `checkForUpdates()`
  for the manual button. Instantiated once by `VibePulseApp` and held for
  the app lifetime.
- **Explicit on-launch check (guarded).** After the updater is started,
  `UpdaterController.init` inspects `updater.automaticallyChecksForUpdates`
  and only calls `updater.checkForUpdatesInBackground()` when that
  property is `true`. `automaticallyChecksForUpdates` is Sparkle's
  persisted state: it is initialized from the `SUEnableAutomaticChecks`
  Info.plist default on first launch and can be toggled at runtime (by
  Sparkle's built-in settings UI, or by a future VibePulse preference
  that wraps it). Calling `checkForUpdatesInBackground()`
  unconditionally would override a user who had disabled automatic
  checks, so the guard is a precondition for any future
  "automatically check for updates" toggle. When the guard passes,
  the explicit call forces a silent background check shortly after app
  launch rather than relying solely on Sparkle's timer-based overdue
  detection. The behavior is harmless when nothing is newer (no UI, no
  prompt). Sparkle's 24h scheduler then takes over for subsequent
  checks via `SUEnableAutomaticChecks` + `SUScheduledCheckInterval` in
  Info.plist.
- `MenuContentView.controls` gains a "Check for Updates…" `Button` that
  calls `UpdaterController.checkForUpdates()`.
- `Info.plist` (generated by `build_dmg.sh`) gains four Sparkle keys:
  - `SUFeedURL` — `https://github.com/wesm/vibepulse/releases/latest/download/appcast.xml`
  - `SUPublicEDKey` — base64 Ed25519 public key, literal string
  - `SUEnableAutomaticChecks` — `<true/>`
  - `SUScheduledCheckInterval` — `<integer>86400</integer>`

### Bundle embedding (the risky part)

VibePulse builds via `swift build -c release` + a hand-written
`scripts/build_dmg.sh`, not Xcode. SwiftPM downloads Sparkle as a binary
target into `.build/artifacts/.../Sparkle.xcframework`. Unlike Xcode, `swift
build` does not embed frameworks into an app bundle for us.

`build_dmg.sh` gains a post-build step that:

1. Locates the macOS slice of the xcframework at
   `.build/artifacts/<pkg-id>/Sparkle/Sparkle.xcframework/macos-arm64_x86_64/Sparkle.framework`.
   The exact parent path varies with SwiftPM internals, so the script
   resolves it via `find .build/artifacts -type d -path '*macos-arm64_x86_64/Sparkle.framework'`
   and fails fast if zero or multiple matches are returned.
2. Copies `Sparkle.framework` → `VibePulse.app/Contents/Frameworks/Sparkle.framework`
   with `cp -R` preserving symlinks.
3. **Fixes the executable's runtime search path.** `swift build` produces
   a Mach-O binary whose load commands reference Sparkle via its
   absolute `.build/artifacts/...` path rather than
   `@executable_path/../Frameworks`. Signing-only would pass, but the
   bundle would fail to launch on any machine other than the build
   machine because `dyld` cannot find the framework at the original
   absolute path. Two complementary fixes:
   1. **Preferred — rewrite via `install_name_tool`.** After copying the
      executable into the bundle but before signing:
      ```
      install_name_tool -add_rpath "@executable_path/../Frameworks" \
          "$APP_DIR/Contents/MacOS/$APP_NAME"
      install_name_tool -change "<sparkle_absolute_install_name>" \
          "@rpath/Sparkle.framework/Versions/B/Sparkle" \
          "$APP_DIR/Contents/MacOS/$APP_NAME"
      ```
      The exact old install name is discovered at build time with
      `otool -L "$APP_DIR/Contents/MacOS/$APP_NAME" | grep Sparkle | awk '{print $1}'`.
      `install_name_tool` modifies the binary, which invalidates any
      existing signature — that is fine because this runs before the
      signing step below.
   2. **Backup — Package.swift linker flag.** If the install_name_tool
      approach turns out to be brittle across Sparkle releases,
      `Package.swift` can instead declare
      `linkerSettings: [.unsafeFlags(["-Xlinker", "-rpath", "-Xlinker",
      "@executable_path/../Frameworks"])]` on the executable target so
      the rpath is baked into the binary at link time. This avoids the
      post-build rewrite but pulls an `unsafeFlags` call into the
      manifest.
4. **Verifies the fix before signing.** `otool -L
   "$APP_DIR/Contents/MacOS/$APP_NAME" | grep Sparkle` must show the
   Sparkle reference starting with `@rpath/` (or `@executable_path/`),
   not an absolute path. `otool -l "$APP_DIR/Contents/MacOS/$APP_NAME"
   | grep -A2 LC_RPATH` must contain
   `@executable_path/../Frameworks`. If either check fails, the script
   aborts with a clear error. This is the step that catches silent
   search-path regressions that signing and notarization do not.
5. Re-signs nested components in strict dependency order before signing
   the outer app:
   1. Each XPC service under
      `Sparkle.framework/Versions/B/XPCServices/*.xpc` individually
      (typically `Downloader.xpc` and `Installer.xpc`).
   2. The nested `Sparkle.framework/Versions/B/Autoupdate.app` helper app
      (signs its own Mach-O, not as a bundle container).
   3. `Sparkle.framework` itself (the Versions/B symlink target).
   4. The outer `VibePulse.app` bundle last.
   Each `codesign` call uses `--options runtime --timestamp --sign
   "$SIGN_IDENTITY"`. The existing `--deep` flag is dropped from signing —
   it is deprecated and does not correctly handle Sparkle's nested
   helpers.
6. Final verification: `codesign --verify --deep --strict --verbose=2
   VibePulse.app`. The `--deep` flag is still allowed on verification;
   only signing requires explicit per-component ordering.

### Server (release workflow)

The appcast references GitHub Release metadata (the release body, the
stapled DMG's final download URL) that does not exist until the Release
object has been created. A naïve "create release, then attach appcast"
ordering would publish the new release as latest before its
`appcast.xml` asset is attached, opening a 10–30 second window during
which any Sparkle client that checks for updates hits a 404 on
`releases/latest/download/appcast.xml`. To keep the feed continuously
valid, the new release is created as a **draft** first, populated with
all assets including the appcast, then flipped to published as the
last step. The previous release remains "latest" throughout; the new
release only becomes latest when it is already fully assembled. The
workflow therefore runs in **three phases**:

**Phase 1 — Create draft release with core artifacts:**

1. Existing "Build signed DMG" step (now with Sparkle embedded and
   rpath-rewritten) runs.
2. Existing "Notarize DMG" + "Verify notarization" steps unchanged.
3. Existing "Generate checksums" step unchanged.
4. "Upload to GitHub Release" step runs, creating a **draft**
   release containing the DMG and `SHA256SUMS`. The
   `softprops/action-gh-release` step is updated with two inputs:
   - `draft: true` — the release is not yet visible as latest to
     Sparkle clients. `releases/latest` continues to resolve to the
     previous published release and its already-attached
     `appcast.xml`.
   - `generate_release_notes: true` — GitHub auto-populates the draft
     Release body from the commit range since the previous tag.
     Without this, the current workflow at
     `.github/workflows/release.yml` would leave the body empty and
     Phase 2 would always hit the stub fallback; local releases via
     `release_local.sh` (which passes `--generate-notes` to
     `gh release create`) would have real notes while CI releases
     would not. Adding `generate_release_notes: true` aligns the two
     paths.
   `gh release view --json body` reads draft releases, so the body
   populated here is available to Phase 2.

**Phase 2 — Generate and attach appcast (release still draft):**

5. **New step.** `scripts/generate_appcast.sh <version>`:
   1. Computes DMG SHA-256 and size in bytes.
   2. Invokes `Sparkle.framework/Versions/B/Resources/sign_update <dmg>`
      with the private key from the new GH secret
      `SPARKLE_ED_PRIVATE_KEY`. The secret is written to a temp file at
      step start and deleted in the job cleanup step.
   3. Fetches the release body from the draft Release:
      `gh release view "v$VERSION" --json body --jq .body`. Because
      Phase 1 created the draft with `generate_release_notes: true`,
      this call returns the auto-generated notes.
   4. Converts markdown → HTML via
      `gh api -X POST /markdown -F text=@- < body.md`. The `-F`
      (`--field`) flag is a typed parameter that interprets `@-` as
      "read from stdin" and JSON-encodes the body content into the
      `text` field. The `-f` (`--raw-field`) flag does **not** do
      this — it would send the literal string `"@-"` as the field
      value — so the script must use `-F`. On `gh api` failure or an
      empty body, the script falls back (in order) to wrapping the
      raw markdown in `<pre>` CDATA, then to the stub string
      `"Release v$VERSION"`. Never blocks a release on HTML
      rendering.
   5. Emits `dist/appcast.xml` from a single-item template (see Data
      flow below).
   6. Validates XML with `xmllint --noout dist/appcast.xml`.
   7. Greps for `sparkle:edSignature` to fail fast on a missing sig.
6. **New step.** `gh release upload "v$VERSION" dist/appcast.xml
   --clobber` attaches the appcast to the still-draft Release. The
   `--clobber` flag lets CI reruns of Phase 2 replace a
   previously-uploaded appcast rather than failing on "asset already
   exists" (matches the existing local re-upload behavior in
   `release_local.sh`).

**Phase 3 — Publish the draft atomically:**

7. **New step.** `gh release edit "v$VERSION" --draft=false` flips
   the draft to a published release. At this instant, the new
   release atomically becomes "latest" for GitHub's `releases/latest`
   endpoint, and both the DMG and the appcast are already attached.
   Any Sparkle client checking for updates either sees the previous
   release's feed (before this step) or the new release's feed
   (after this step), never a 404. This is the step that closes
   the feed-consistency window.

### Release-notes source

The spec uses the GitHub Release body as the source for Sparkle's rendered
release notes. This keeps a single source of truth that the user can edit
on github.com after the fact (at the cost of potentially drifting from
what Sparkle shows — appcast is a one-shot asset and does not
retroactively update after edits). If preserving an editable release body
turns out to matter less than having release notes match exactly what
Sparkle shows, a future change can switch the source to an annotated tag
message (`git for-each-ref refs/tags/v$VERSION --format='%(contents)'`),
which is available from the moment of tag push and removes the Phase 1 /
Phase 2 ordering dependency entirely.

### Local release flow (`scripts/release_local.sh`)

The local flow is not retired by this spec. It must also produce a
Sparkle-compatible release without opening a feed-consistency window
during which `releases/latest/download/appcast.xml` would return 404.
The local flow therefore adopts the same draft-then-publish ordering
as CI:

1. `scripts/release_local.sh` continues to call `scripts/build_dmg.sh`,
   which now embeds, rpath-rewrites, and re-signs Sparkle.framework.
   No additional work — embedding is shared between local and CI
   paths.
2. Release publication becomes three-phase. The current script's
   `gh release create "$VERSION_RAW" "$DMG_PATH" --generate-notes`
   gains a `--draft` flag. The else-branch (when the release already
   exists) uses `gh release upload --clobber` as before; the
   if-branch creates a new draft release with DMG attached.
3. After either branch completes, the script runs
   `scripts/generate_appcast.sh "$VERSION_RAW"` (which picks the
   private key from the login Keychain locally) and then
   `gh release upload "$VERSION_RAW" dist/appcast.xml --clobber`.
4. Finally, `gh release edit "$VERSION_RAW" --draft=false` publishes
   the draft, atomically making the new release "latest" with both
   DMG and appcast already attached. If the release was already
   published (re-upload path), the `--draft=false` call is a no-op
   and the script ignores a "release is not a draft" error.

The same three-phase ordering as CI. The two scripts stay in sync on
the release-visibility invariant.

### Key management

- Generate the Ed25519 keypair once, locally, with
  `Sparkle.framework/Versions/B/Resources/generate_keys`. The tool writes
  the private key into the login Keychain and prints the base64 public
  key to stdout.
- **Private key** → extract from the login Keychain to a file:
  `security find-generic-password -s "https://sparkle-project.org" -w >
  sparkle-ed-private.txt`. Upload to GH secret: `gh secret set
  SPARKLE_ED_PRIVATE_KEY < sparkle-ed-private.txt`. Local backup alongside
  the Apple credentials in `~/code/agentsview-release-secrets/sparkle-ed-private.txt`.
  Delete the extracted file from the working copy after upload.
- `generate_appcast.sh` picks its private-key source based on
  environment:
  - **In CI** — `SPARKLE_ED_PRIVATE_KEY` is set from the GH secret. The
    script writes it to a temp file, calls `sign_update -f <path>`, and
    deletes the temp file in the job's cleanup step (success or
    failure).
  - **Locally (release_local.sh)** — `SPARKLE_ED_PRIVATE_KEY` is
    typically unset. The script falls through to `sign_update <dmg>`
    with no `-f`, which uses Sparkle's default behavior of reading the
    private key from the login Keychain (where `generate_keys`
    originally stored it). If a developer does set
    `SPARKLE_ED_PRIVATE_KEY` locally — e.g. to test the CI code path —
    the temp-file branch runs instead.
- **Public key** → committed to the repo as a literal string in
  `scripts/build_dmg.sh` where the Info.plist is generated. Public by
  design; baked into every build.

#### Key rotation

Sparkle supports two rotation paths. The spec uses the first when
possible and falls back to the second only under duress.

**Planned rotation (old key still available):** ship a transition release
that is signed with the **old** private key but whose Info.plist contains
the **new** `SUPublicEDKey`. Clients on the previous version verify this
release's enclosure against their existing (old) public key, install
normally, and now carry the new public key. Subsequent releases signed
with the new key then verify cleanly for anyone who picked up the
transition release.

However, this path has a window-of-exposure limitation that flows from
the single-item-appcast design (see *Non-goals* above). The appcast only
advertises the current release. As soon as the next release after the
transition ships, the transition release's item is no longer in the feed,
and any user who did not pick it up during that window will see only the
new-key-signed release in the feed and fall into the compromised-key
recovery path below — Sparkle refuses the new enclosure because their
embedded public key still matches the old one, and they must manually
reinstall.

For a menubar app checking on the default 24h cadence with a user who
launches their Mac daily, the window is typically short (one release
cycle). For users who are offline for extended periods, or if releases
are shipped in rapid succession, the window can be missed. Routine
rotations should therefore be announced out-of-band (release notes, brief
pause between the transition release and the next release) to give users
a chance to check in.

The fully-safe version of this path requires migrating to a multi-item
aggregated appcast before rotating, so the transition release stays
visible in the feed for weeks or months. That migration is in *Open
questions* as an option deferred until needed. Until then, routine
rotation remains possible but depends on users checking during the
transition window.

Steps for a routine rotation using the single-item appcast as-is:
1. Generate a new keypair (`generate_keys`).
2. Set the new public key as `SUPublicEDKey` in `scripts/build_dmg.sh`.
3. Build the next release, but temporarily pass the **old** private key
   to `sign_update` via the existing `SPARKLE_ED_PRIVATE_KEY` secret.
4. Ship. Announce in the release notes that users should allow the
   automatic update to run.
5. Wait a rotation-window buffer (at minimum one full scheduled-check
   cycle, realistically a couple of weeks for stragglers).
6. Update the `SPARKLE_ED_PRIVATE_KEY` secret to the new private key.
   Subsequent releases use the new key pair throughout. Users who did
   not pick up the transition release during the window must manually
   reinstall.
7. Document the rotation in `docs/github-actions-release.md`.

**Compromised or lost key (old key unavailable):** generate a new pair,
update `SUPublicEDKey` and `SPARKLE_ED_PRIVATE_KEY`, ship. Every existing
user will see Sparkle refuse the update because the new EdDSA signature
cannot be verified against their embedded (old) public key. These users
must manually download and install the new DMG once from GitHub Releases.
From there, subsequent updates resume normally. This path is acceptable
only for incident response, not for routine rotation.

## Data flow

### Release time (CI, on `git push origin v0.3.0`)

1. `release.yml` runs existing cert import + notary key write steps.
2. `scripts/build_dmg.sh v0.3.0` runs:
   - `swift build -c release` builds the executable.
   - Executable copied into a fresh `VibePulse.app` bundle.
   - Icon + generated Info.plist copied in (Info.plist now includes
     the four Sparkle keys).
   - **New:** Sparkle.framework located in SwiftPM artifacts and copied
     into `VibePulse.app/Contents/Frameworks/Sparkle.framework`.
   - **New:** `install_name_tool` rewrites the executable's Sparkle
     install name to `@rpath/Sparkle.framework/Versions/B/Sparkle` and
     adds an `LC_RPATH` entry pointing at
     `@executable_path/../Frameworks`. `otool -L` and `otool -l`
     verify the result; script aborts if the references are still
     absolute `.build/...` paths.
   - **New:** Per-component sign (XPC services → Autoupdate.app →
     framework → app) with Hardened Runtime + timestamp.
   - DMG built via `hdiutil`.
3. `notarytool submit --wait` + `stapler staple` — unchanged.
4. `spctl --assess` mounted-DMG check — unchanged.
5. `shasum -a 256 *.dmg > SHA256SUMS` — unchanged.
6. **Phase 1 upload** (`softprops/action-gh-release` with `draft: true`
   and `generate_release_notes: true`) — creates a **draft** Release
   for tag `v0.3.0` with `dist/*.dmg` and `dist/SHA256SUMS` attached
   and body populated from GitHub's auto-generated notes. The draft
   is not visible to `releases/latest`; the previous published release
   remains latest throughout Phases 1 and 2.
7. **New:** `scripts/generate_appcast.sh v0.3.0` runs. The draft
   release's body is read via `gh release view "v0.3.0" --json body`
   (draft releases are readable by the repo owner). Emits
   `dist/appcast.xml`.
8. **New:** `gh release upload "v0.3.0" dist/appcast.xml --clobber`
   attaches the appcast to the still-draft Release (idempotent on
   rerun via `--clobber`).
9. **New:** `gh release edit "v0.3.0" --draft=false` atomically
   publishes the draft. At this instant `releases/latest` flips from
   the previous release to this one, with DMG and appcast both
   already attached. Sparkle clients never observe a release with a
   missing appcast asset.

### Appcast XML shape

```xml
<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0" xmlns:sparkle="http://www.andymatuschak.org/xml-namespaces/sparkle">
  <channel>
    <title>VibePulse</title>
    <link>https://github.com/wesm/vibepulse</link>
    <description>VibePulse release feed</description>
    <language>en</language>
    <item>
      <title>Version 0.3.0</title>
      <sparkle:version>0.3.0</sparkle:version>
      <sparkle:shortVersionString>0.3.0</sparkle:shortVersionString>
      <sparkle:minimumSystemVersion>13.0</sparkle:minimumSystemVersion>
      <pubDate>Mon, 13 Apr 2026 15:25:00 +0000</pubDate>
      <description><![CDATA[<!-- HTML rendered from GitHub release body -->]]></description>
      <enclosure
        url="https://github.com/wesm/vibepulse/releases/download/v0.3.0/VibePulse-0.3.0.dmg"
        sparkle:edSignature="..."
        length="275148"
        type="application/octet-stream" />
    </item>
  </channel>
</rss>
```

### Client time (running VibePulse on a user's Mac)

1. App launches. `UpdaterController` instantiates
   `SPUStandardUpdaterController(startingUpdater: true, updaterDelegate:
   nil, userDriverDelegate: nil)`. Sparkle reads `Info.plist` keys
   (`SUFeedURL`, `SUPublicEDKey`, `SUEnableAutomaticChecks`,
   `SUScheduledCheckInterval`) on startup. After startup,
   `UpdaterController` checks `updater.automaticallyChecksForUpdates`
   and calls `updater.checkForUpdatesInBackground()` only if that
   property is `true`, respecting any persisted user preference that
   overrides the Info.plist default.
2. The on-launch background check from step 1 (plus Sparkle's
   subsequent 24h scheduler for long-running sessions) issues HTTPS GET
   `https://github.com/wesm/vibepulse/releases/latest/download/appcast.xml`
   — GitHub 302-redirects to the newest non-draft, non-prerelease
   release's asset.
3. Sparkle parses the RSS and compares `sparkle:version` against the
   app's `CFBundleVersion` (the technical version — Sparkle's primary
   comparison axis, separate from the marketing
   `CFBundleShortVersionString` which is shown to the user but not used
   for ordering). If `sparkle:shortVersionString` is present it is used
   for display but not for comparison. `scripts/build_dmg.sh` currently
   sets both `CFBundleVersion` and `CFBundleShortVersionString` to the
   same release-tag value; the appcast's `sparkle:version` must match
   that technical value, and this spec treats `CFBundleVersion` as the
   canonical comparison key. If the two plist fields ever diverge
   (e.g. switching to a monotonic build-number scheme), only the
   `CFBundleVersion`-derived field in the appcast affects update
   decisions. If newer, proceeds.
4. Sparkle downloads the DMG to a temp cache, verifies the EdDSA
   signature against the embedded `SUPublicEDKey`, verifies the Apple code
   signature and notarization staple. Any failure aborts with a user-facing
   error and no install.
5. The standard Sparkle dialog displays with rendered release notes and
   Install / Later / Skip buttons.
6. User clicks **Install Update**. Sparkle hands the verified DMG to its
   `Autoupdate.app` helper, which mounts the DMG, copies `VibePulse.app`
   over the existing install, and relaunches.

### Manual-check path

"Check for Updates…" in the menubar popover calls
`updater.checkForUpdates()` (user-initiated API). Jumps to step 3 in the
client flow above but surfaces "You're up to date" if no newer version is
found, rather than failing silently.

### First-version caveat

Users currently on v0.2.0 or earlier have no Sparkle framework bundled, so
they must manually install the first Sparkle-enabled release. After that,
all subsequent updates are in-app.

### `releases/latest` semantics

GitHub's `/releases/latest/download/<asset>` endpoint resolves to the
newest release that is **not a draft and not a prerelease**. Two
consequences this spec relies on:

1. **Prerelease tags are invisible.** Tagging anything like
   `v0.3.0-beta.1` is therefore invisible to Sparkle clients by
   design, matching the "no prerelease channel" non-goal.
2. **Draft releases are invisible until published.** The three-phase
   workflow uses this to keep the feed continuously valid across
   releases: the new release is created as a draft in Phase 1 and
   flipped to published in Phase 3, so `releases/latest` jumps
   directly from "previous release with old appcast" to "new
   release with new appcast" without ever pointing at a half-
   assembled release.

## Error handling

### Client side (handled by Sparkle; documented only)

- **Network error / feed unreachable** — Sparkle retries on next scheduled
  check. Manual check shows a native error dialog.
- **Malformed appcast XML** — Sparkle logs to
  `~/Library/Logs/Sparkle/` and reports no update. Manual check surfaces
  a parse error.
- **EdDSA signature mismatch** — Sparkle refuses to install, shows a
  fatal error dialog. This is the primary protection against a tampered
  GitHub Release asset. Fails closed.
- **Notarization staple missing or invalid** — Sparkle's `SPUDownloadedUpdate`
  validation rejects the DMG before handing it to `Autoupdate.app`. Also
  fails closed.
- **Install fails mid-replace** — `Autoupdate.app` runs as a separate
  process, keeps the original bundle as a backup, rolls back on failure.
  User sees an error; the previous version keeps running.
- **User on macOS < 13** — `<sparkle:minimumSystemVersion>13.0</sparkle:minimumSystemVersion>`
  hides the update from unsupported clients.

### CI side (we own)

- **`sign_update` can't read private key** — bash `set -euo pipefail`
  aborts; GH Actions job fails before upload. Recovery: re-check
  `SPARKLE_ED_PRIVATE_KEY` secret content.
- **`gh api /markdown` fails or rate-limits** — `generate_appcast.sh`
  falls back to `<pre>`-wrapped raw markdown, then to `"Release v$VERSION"`
  as a stub. Never blocks a release on release-notes rendering.
- **Embedding step fails** (Sparkle.xcframework not found at expected
  path) — `build_dmg.sh` aborts with a clear error pointing at the
  SwiftPM artifacts path. Caught locally before CI.
- **rpath rewrite fails** (absolute `.build/...` path still present
  after `install_name_tool`) — `build_dmg.sh`'s `otool -L` / `otool -l`
  verification aborts the script before signing. No partially-rewritten
  bundle is ever signed or notarized.
- **Nested signing in wrong order** — `codesign --verify --deep --strict
  --verbose=2` at the end of `build_dmg.sh` catches this. The existing
  `spctl --assess` CI step provides a second line of defense after
  notarization.
- **Phase 3 publish fails** (job errors out between `gh release upload`
  of appcast and `gh release edit --draft=false`) — the draft release
  remains in the repo with DMG + appcast attached. The previous
  published release is still "latest" and Sparkle clients continue to
  see its feed, so no user impact. Recovery: rerun the release job
  (which re-uploads with `--clobber` and retries the publish), or
  manually run `gh release edit "v$VERSION" --draft=false` against
  the draft. If the job is never retried and the draft lingers, it is
  visible to the repo owner on github.com but not to anonymous users
  or Sparkle, so it is safe to leave or delete.

### No partial-disable flag

The spec does not introduce an `ENABLE_SPARKLE` (or similar) toggle. An
earlier draft proposed gating the shell scripts behind such a flag, but
the client side makes Sparkle a hard SwiftPM dependency and instantiates
`SPUStandardUpdaterController` at app launch for the app's lifetime.
Disabling only the shell-script half would leave a client binary that
expects the framework to be embedded and an Info.plist that advertises a
feed URL, producing runtime errors rather than a clean pre-Sparkle
release. Half-gating is worse than no gate.

If updater plumbing ever blocks a release, the recovery path is to
revert the Sparkle-integration commit(s) on a hotfix branch, tag from
there, and ship. This is a coarser but honest escape hatch that does not
leave a misleading half-disabled code path in the main branch.

## Testing

### Local pre-merge

- **Build smoke** — `swift build` succeeds with the new Sparkle SPM
  dependency resolved and linked.
- **Bundle inspection** — run `scripts/build_dmg.sh v0.3.0-test` locally:
  - `ls VibePulse.app/Contents/Frameworks/Sparkle.framework/Versions/B/XPCServices`
    shows `Downloader.xpc` and `Installer.xpc`.
  - `otool -L VibePulse.app/Contents/MacOS/VibePulse | grep Sparkle`
    shows `@rpath/Sparkle.framework/Versions/B/Sparkle`, not an
    absolute `.build/...` path. This is the check that catches
    runtime search-path regressions.
  - `otool -l VibePulse.app/Contents/MacOS/VibePulse | grep -A2 LC_RPATH`
    contains `@executable_path/../Frameworks`.
  - `codesign --verify --deep --strict --verbose=2 VibePulse.app` passes.
  - After notarization: `spctl --assess --type execute VibePulse.app`
    passes.
  - **Cold-start smoke test.** Open the built `.app` by double-click
    (not from the build directory's in-place path). If the rpath fix
    is wrong, the app refuses to launch with a "library not loaded"
    dyld error even though signing and notarization passed. This
    catches embedding bugs that no static check can.
- **Appcast generation dry-run** — a local-only tag will not have a
  GitHub Release object, so `generate_appcast.sh` cannot fetch a release
  body via `gh release view`. For local testing, the script accepts an
  override environment variable `APPCAST_RELEASE_BODY_FILE` pointing at
  a markdown file on disk; when set, the script reads the body from
  that file instead of calling `gh release view`. Run:
  `APPCAST_RELEASE_BODY_FILE=/tmp/stub.md scripts/generate_appcast.sh
  v0.3.0-test`, then `xmllint --noout dist/appcast.xml`. The override
  is only respected when set explicitly; in CI the variable is unset
  and the normal `gh release view` path runs.
- **EdDSA signature round-trip** — use `sign_update -p <pubkey> <sig>
  <dmg>` (verification mode) to confirm the signature validates against
  the committed public key. Catches key-mismatch before publishing.

### End-to-end

Sparkle's full update path can only be exercised against a real published
release pair. Plan:

1. **v0.3.0** — ships with Sparkle framework embedded but is the last
   release users must install manually. No prior version has Sparkle, so
   this is unavoidable.
2. **v0.3.1** — trivial change (e.g. copyright-year bump). Install v0.3.0
   locally, trigger a manual update check, click through the Sparkle
   dialog, verify v0.3.1 installs and relaunches cleanly. This is the real
   test.
3. Only after v0.3.0 → v0.3.1 succeeds is the feature considered validated.

### CI per-release

- `xmllint --noout dist/appcast.xml` fails the job on malformed XML.
- `grep sparkle:edSignature dist/appcast.xml` fails the job if the
  signature attribute is missing.
- `otool -L "$APP_DIR/Contents/MacOS/$APP_NAME" | grep -q '@rpath/Sparkle.framework'`
  fails the build if the rpath rewrite step did not apply.
- Existing `spctl --assess` step catches embedding regressions after
  notarization.

### What is NOT tested

- Sparkle internals (download, signature verification, XPC lifecycle,
  install/relaunch). Trusted upstream.
- `UpdaterController` — ~20-line adapter with no branching logic worth a
  unit test.
- `generate_appcast.sh` — a shell script with no pure logic; validated
  by its output (xmllint + grep in CI).

## Open questions

None requiring decisions before implementation. The following are
deferred until they become real problems:

- Multi-item appcast / aggregated history (migrate to GitHub Pages if
  version-skipping becomes common).
- Beta channel (add a second appcast URL, gated by a hidden defaults
  key).
- Delta updates (only relevant if the DMG grows substantially).

### Resolved in review

- **Is `scripts/release_local.sh` being retired?** No. It continues to
  exist and now also produces Sparkle-compatible releases: `build_dmg.sh`
  does the framework embedding (shared with CI), and
  `release_local.sh` calls `generate_appcast.sh` after the Release is
  created, then uploads the appcast with `gh release upload --clobber`.
  See *Local release flow* above.
- **Does Sparkle compare `CFBundleVersion` or
  `CFBundleShortVersionString`?** `CFBundleVersion`. The spec previously
  said `CFBundleShortVersionString`, which worked only because
  `scripts/build_dmg.sh` currently sets both plist fields to the same
  value. Corrected in the client-time data flow above. If
  `build_dmg.sh` ever starts setting them to different values,
  `sparkle:version` in the appcast must match `CFBundleVersion`, not
  the marketing string.

## Deferred implementation-time checks

Items the implementer should verify at the start of the plan rather than
committing to in this spec:

- **Sparkle 2.9+ inline markdown release notes.** Recent Sparkle releases
  may support inline markdown in the `<description>` element (or via a
  different content-type hint), which would remove the need for the
  `gh api /markdown` HTML conversion step. If available in the pinned
  Sparkle version, the appcast template can embed the markdown body
  directly and `generate_appcast.sh` drops the conversion call. If not
  available, the HTML-conversion path stays as specified. Decide after
  looking at the Sparkle documentation for the specific version pinned
  in `Package.swift`.
- **Exact `sign_update` file flag.** The spec uses `-f <path>` as the
  likely flag for loading the private key from a file; the implementer
  should confirm the current flag name against `sign_update --help` in
  the embedded Sparkle version and adjust if it differs. This does not
  change any design decision, only the command syntax in the script.
